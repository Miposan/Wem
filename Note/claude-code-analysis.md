# Claude Code 泄露源码架构分析

## 1. 泄露事件

| 项 | 值 |
|---|---|
| 时间 | 2026 年 3 月 31 日 |
| 原因 | npm 包中包含未剥离的 source map（59.8 MB） |
| 规模 | ~512,000 行 TypeScript，~1,900 个文件 |
| 影响范围 | 完整源码，包括系统提示、工具实现、安全机制、隐藏功能 |
| 官方回应 | Anthropic 未正式确认，但后续版本移除了 source map |

这不是一次"开源"，而是一次意外泄露。但它提供了目前最完整的
生产级 AI Agent 架构参考。

## 2. 整体架构：三层模型

Claude Code 的架构可以用三层来描述：

```
┌──────────────────────────────────────────────────────────┐
│                      UI Layer                            │
│                                                          │
│   Terminal UI (Ink/React)                                │
│   ├─ 消息渲染（Markdown、代码高亮、diff 视图）              │
│   ├─ 权限确认弹窗（approve / deny）                        │
│   ├─ 工具调用实时展示（spinner + 结果）                     │
│   └─ 输入框（多行、自动补全、/命令）                        │
│                                                          │
│   IDE Extensions (VS Code / JetBrains)                   │
│   ├─ Language Server Protocol                            │
│   ├─ 内联 diff 预览                                      │
│   └─ 侧边栏对话面板                                      │
├──────────────────────────────────────────────────────────┤
│                    Harness Layer                          │
│                                                          │
│   ┌────────────┐  ┌──────────┐  ┌───────────────────┐  │
│   │  Agentic   │  │ Tool     │  │ Permission        │  │
│   │  Loop      │  │ Registry │  │ System            │  │
│   └─────┬──────┘  └────┬─────┘  └───────────────────┘  │
│         │              │                                  │
│   ┌─────┴──────────────┴──────────────────────────────┐  │
│   │              ~40 Built-in Tools                    │  │
│   │  Read │ Edit │ Write │ Bash │ Glob │ Grep │ Agent │  │
│   │  WebFetch │ NotebookEdit │ TodoWrite │ ...        │  │
│   └───────────────────────────────────────────────────┘  │
│                                                          │
│   ┌───────────────────────────────────────────────────┐  │
│   │           Memory & Context                        │  │
│   │  memory.md │ CLAUDE.md │ grep search │ tokenizer  │  │
│   └───────────────────────────────────────────────────┘  │
│                                                          │
│   ┌───────────────────────────────────────────────────┐  │
│   │           Prompt Assembly                         │  │
│   │  System Prompt 模块化拼接：                         │  │
│   │  base + tools + rules + memory + CLAUDE.md + ...  │  │
│   └───────────────────────────────────────────────────┘  │
├──────────────────────────────────────────────────────────┤
│                    Model Layer                            │
│                                                          │
│   Anthropic Messages API（专有，硬绑定）                   │
│   ├─ 流式响应（SSE）                                      │
│   ├─ tool_use / tool_result 消息块                       │
│   └─ extended_thinking（思维链）                           │
│                                                          │
│   未来: 多模型支持（代码中发现 Gemini 集成痕迹）            │
└──────────────────────────────────────────────────────────┘
```

**核心洞察**：Claude Code 的护城河不在 Model 层（任何人都调得了 API），
而在 **Harness 层**——工具编排、权限控制、上下文管理、提示组装。

## 3. Agentic Loop（编排核心）

### 3.1 基本循环

Claude Code 的核心是一个 `while(true)` 循环：

```
用户输入
    │
    ▼
┌─────────────────────────────────┐
│  while (step < max_steps) {     │  ← max_steps = 1000
│                                  │
│    1. 组装 System Prompt         │
│       + 消息历史                  │
│       + 工具定义                  │
│                                  │
│    2. 调用 LLM (streamText)      │  ← 流式
│       │                          │
│       ├─ 文本 → 渲染给用户        │
│       └─ tool_use → 解析参数      │
│                                  │
│    3. 对每个 tool_use:            │
│       ├─ 权限检查                 │
│       ├─ 执行工具                 │
│       └─ 结果追加为 tool_result   │
│                                  │
│    4. 消息列表追加本轮结果         │
│       → 回到第 1 步              │
│  }                               │
│                                  │
│  输出最终文本给用户               │
└─────────────────────────────────┘
```

### 3.2 关键设计决策

| 决策 | 说明 |
|------|------|
| **流式优先** | LLM 响应是 SSE 流，文本 delta 实时推到终端 |
| **批量 tool_calls** | LLM 一次可返回多个 tool_use，并行执行读操作 |
| **步数上限 1000** | 比一般 Agent 高得多，说明 Claude Code 预期复杂任务 |
| **错误不中断** | 工具失败 → tool_result(is_error=true) → LLM 自行调整 |
| **无递归深度限制** | 子 Agent 本质是新 Loop，共享相同的 max_steps |

### 3.3 streamText 内部流程

```
HTTP POST /v1/messages
    │
    ├─ system: 拼接后的 System Prompt（可长达数万 token）
    ├─ messages: 完整对话历史
    ├─ tools: ~40 个工具的 JSON Schema 定义
    ├─ stream: true
    │
    ▼
SSE 事件流:
    message_start     → 获取 message ID、模型信息
    content_block_start → 开始一个 content block
    content_block_delta → 文本 delta 或 tool_use 参数 delta
    content_block_stop  → 结束一个 content block
    message_stop      → 整体结束
```

## 4. 工具系统

### 4.1 ~40 个内置工具一览

| 分类 | 工具 | 说明 |
|------|------|------|
| **文件操作** | `Read` | 读文件（支持 offset/limit 分页） |
| | `Edit` | 精确编辑（str replace 模式） |
| | `Write` | 写入/创建文件 |
| **搜索** | `Glob` | 文件名 glob 匹配 |
| | `Grep` | 文本搜索（支持 regex） |
| **执行** | `Bash` | Shell 命令执行（2500 行安全检查） |
| **网络** | `WebFetch` | 抓取 URL 内容 |
| **代理** | `Agent` | 子 Agent 派生（探索、并行研究） |
| **笔记本** | `NotebookEdit` | Jupyter notebook 编辑 |
| **任务管理** | `TodoWrite` | 内部 todo 列表 |
| **规划** | `PlanTool` | 创建/更新执行计划 |
| **计划任务** | `CronCreate/Delete/List` | 定时任务 |
| **外部唤醒** | `ScheduleWakeup` | 外部事件唤醒 |
| **用户交互** | `AskUserQuestion` | 向用户提问 |
| **搜索** | `WebSearch` | 网络搜索 |
| **代码分析** | `ToolSearch` | 动态加载延迟工具 |
| **工作树** | `EnterWorktree/ExitWorktree` | Git worktree 隔离 |
| **MCP** | MCP 客户端 | 外部工具协议 |

### 4.2 工具定义模式

每个工具是一个 TypeScript 类，包含：

| 属性 | 说明 |
|------|------|
| `name` | 工具名（如 `bash`、`read`） |
| `description` | 给 LLM 看的功能描述 |
| `inputSchema` | JSON Schema 参数定义 |
| `prompt` | 使用指南（注入到 System Prompt） |

工具的 prompt 是一个重要的架构决策——
LLM 不是仅靠 JSON Schema 理解工具，而是有详细的使用指南。

### 4.3 Bash 工具的安全系统

Bash 工具有 **2500 行**的安全检查代码，是最大的单个工具实现：

```
Bash 工具安全检查流程:
    │
    ├─ 1. 命令解析（支持管道、链式、子 shell）
    ├─ 2. 危险命令检测
    │   ├─ rm -rf /                → 直接拒绝
    │   ├─ git push --force        → 需确认
    │   ├─ 环境变量修改             → 需确认
    │   └─ 网络操作（curl/wget）    → 需确认
    ├─ 3. 沙箱选项
    │   ├─ Docker 容器隔离
    │   ├─ 网络隔离模式
    │   └─ 只读文件系统
    └─ 4. 超时控制
        ├─ 默认 120s 超时
        └─ 长时间任务提示用户
```

### 4.4 子 Agent（Agent 工具）

子 Agent 是 Claude Code 实现多 Agent 的方式：

```
主 Agent
    │
    ├─ Agent(subagent_type="explore")
    │   用途: 广泛探索代码库，3+ 次查询时自动建议使用
    │   特点: 独立上下文窗口，结果摘要返回
    │
    └─ Agent(subagent_type="default")
        用途: 通用子任务委托
        特点: 完整工具集，独立 Loop
```

**关键设计**：子 Agent 不是框架级抽象，而是通过 **prompt** 控制行为。
LLM 被告知"你是一个子 Agent，完成后返回结果"——
行为差异完全来自 System Prompt，而非不同的代码路径。

```
┌─────────────────────────────────────────────────────────┐
```

### 5.1 Memory 文件的组织

```
~/.claude/projects/{project-hash}/memory/
├── MEMORY.md          ← 索引文件，每次对话自动加载
├── user_role.md       ← 用户画像（角色、技能水平）
├── feedback_testing.md ← 行为反馈（"不要 X"，"继续 Y"）
├── project_context.md  ← 项目上下文（里程碑、决策）
└── reference_links.md  ← 外部资源指针
```

**MEMORY.md 是索引**，不是记忆本身。它被自动加载到对话上下文中，
让 Agent 知道有哪些记忆文件可供查询。实际的记忆内容存在各主题文件中，
Agent 按需读取（grep 或 Read）。

### 5.2 记忆的写入时机

| 触发 | 写入内容 |
|------|---------|
| 用户明确要求记住 | 写入对应主题文件 |
| 用户纠正行为 | 写入 feedback 类型 |
| 发现用户角色/偏好 | 写入 user 类型 |
| 项目里程碑/决策 | 写入 project 类型 |
| 学习到外部资源 | 写入 reference 类型 |

## 6. System Prompt 组装

Claude Code 的 System Prompt 是模块化拼接的，不是一整段固定文本：

```
最终 System Prompt
    │
    ├── 基础指令（你是 Claude Code，Anthropic 的官方 CLI）
    │   ├── 角色定义
    │   ├── 通用行为规则
    │   └── 输出格式要求
    │
    ├── 工具使用指南（每个工具的 prompt 字段）
    │   ├── Read 工具使用规则
    │   ├── Edit 工具使用规则
    │   ├── Bash 安全规则
    │   ├── Agent 子 Agent 使用指南
    │   └── ... (~40 个工具的指南)
    │
    ├── 项目上下文
    │   ├── CLAUDE.md 内容（项目级配置）
    │   └── .claude/ 目录下的规则文件
    │
    ├── 记忆索引
    │   └── MEMORY.md 内容
    │
    ├── 环境信息
    │   ├── 工作目录
    │   ├── 操作系统
    │   ├── Git 状态（分支、最近提交）
    │   └── 平台信息
    │
    └── 会话上下文
        └── 当前日期时间
```

**Prompt 总量可达数万 token**，其中工具使用指南占最大比例。
这是 Claude Code 的核心知识产权之一。

## 7. 权限系统

### 7.1 三级权限模型

```
工具调用请求
    │
    ▼
权限检查
    │
    ├── Auto    → 直接执行（如 Read 文件、Glob 搜索）
    │
    ├── Ask     → 暂停，通知用户确认
    │             ├─ 用户 Approve → 执行
    │             ├─ 用户 Deny → 返回 "User denied" 给 LLM
    │             └─ 同类操作可记住选择（session 内）
    │
    └── Deny    → 直接拒绝，不询问用户
```

### 7.2 权限配置来源

| 来源 | 优先级 | 说明 |
|------|--------|------|
| 用户本次会话授权 | 最高 | approve 一次后同类操作自动放行 |
| `.claude/settings.json` | 中 | 项目级权限覆盖 |
| 工具默认声明 | 最低 | 每个工具声明自己的默认权限级别 |

### 7.3 与 Wem 的对比

| 维度 | Claude Code | Wem Agent（设计） |
|------|------------|------------------|
| 粒度 | 工具 + 参数级 | 工具 + 参数级 |
| 审批方式 | 终端弹窗 | SSE → 前端弹窗 |
| 持久化 | session 内有效 | 可配置持久规则 |
| 子 Agent 权限 | 继承主 Agent 的限制 | 独立配置 |

这些隐藏功能说明 Claude Code 的架构有很强的**可扩展性**——
新功能可以通过添加模块和配置开关实现，而不需要重构核心。

## 9. Claw-Code：干净室重写

### 9.1 概况

| 项 | 值 |
|---|---|
| 作者 | Sigrid Jin |
| 语言 | Python + Rust |
| 协议 | MIT |
| Stars | 100,000+ |
| 方式 | 干净室重写（不看泄露源码，仅根据公开架构文档） |

### 9.2 重写的架构映射

| Claude Code 原版 | Claw-Code 重写 | 语言 |
|------------------|---------------|------|
| TypeScript + Ink | Python + Textual | UI 层 |

| ~40 工具类 | Python 工具函数 | 工具层 |
| Anthropic 专用 | 多 Provider 支持 | 模型层 |

### 9.3 对 Wem 的启示

Claw-Code 证明了 Claude Code 的架构可以用不同语言重新实现。
Wem 选择 Rust 是合理的——性能更好，类型安全更强。
核心抽象（Loop + Tools + Permission + Context）与语言无关。

## 11. 架构模式总结

### 11.1 Claude Code 的 8 个关键架构模式

| 模式 | 说明 | Wem 适用性 |
|------|------|-----------|
| **Agentic Loop** | while(true) 循环 + 步数上限 | 直接采用 |
| **工具即 Prompt** | 每个工具有详细使用指南，注入 System Prompt | 直接采用 |
| **子 Agent = Prompt** | 通过 prompt 而非代码控制子 Agent 行为 | 直接采用 |
| **三级权限** | Auto / Ask / Deny | 直接采用 |
| **模块化 Prompt** | System Prompt 拼接，不是固定文本 | 直接采用 |
| **文件记忆** | memory/ 目录 + MEMORY.md 索引 | 直接采用 |
| **错误不中断** | 工具失败 → error result → LLM 调整 | 直接采用 |
| **流式优先** | SSE 流式响应，逐字推送给用户 | 直接采用 |

### 11.2 Wem 不需要采用的设计

| 设计 | 原因 |
|------|------|
| DRM / 反蒸馏 | 开源项目不需要 |
| 隐藏功能开关 | 增加复杂度，初期不需要 |
| 1000 步上限 | Wem 场景更简单，50 步足够 |
| 2500 行 Bash 安全检查 | 可简化，Wem 初期可限制为白名单模式 |
| Ink/React 终端 UI | Wem 用 Axum HTTP + SSE，前端自建 |
| 单一 Provider 绑定 | Wem 从设计之初就支持多 Provider |

### 11.3 对 Wem Agent 设计的具体影响

1. **工具定义**：每个工具不仅要有 JSON Schema 参数定义，还要有详细的使用 prompt
2. **System Prompt**：模块化拼接，包含角色 + 工具指南 + 项目上下文 + 记忆索引
3. **记忆系统**：采用 memory/ 目录 + MEMORY.md 索引的模式
4. **子 Agent**：通过 prompt 控制行为差异，不需要独立的代码路径
5. **权限粒度**：Auto/Ask/Deny 三级足够，不需要更复杂的方案
6. **Loop 状态机**：Idle → Thinking → WaitingApproval → Executing → Responding → Idle
